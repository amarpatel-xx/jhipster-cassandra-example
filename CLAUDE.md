# jhipster-cassandra-example — Testing Runbook

A monorepo of **three JHipster microservice apps** generated by the
`generator-jhipster-cassandra` blueprint:

| App | Role | Server port | Cassandra CQL (Docker) port | Entities |
|-----|------|------------|------------------------------|----------|
| `cassandragateway` | gateway (OAuth2, Eureka, microfrontend shell) | **8080** | — (uses Postgres :5432) | user, authority |
| `cassandrablog` | microservice remote | **8081** | 9342 | 12 (Blog, Post, Tag, SaathratriEntity2–4, AddOns*, LandingPage…, SetEntity…, TajUser) |
| `cassandrastore` | microservice remote | **8082** | 9242 | Product, Report (both **single-key**) |

> ⚠️ **Port trap:** `9342`/`9242` are the **Cassandra CQL** ports published by Docker, **not**
> the app servers. The backends listen on **8080 / 8081 / 8082** (`server.port` in each
> `application-dev.yml`). Always health-check the 808x ports, never 934x/924x.

This file is the procedure for running **the full test suite for all three apps**: regen →
backend (unit + IT) → frontend (vitest) → e2e (Cypress). The blueprint's own template-debug
loop lives in `../generator-jhipster-cassandra/TESTING.md` (uses a throwaway sample); this
doc is about the real example monorepo.

> ✅ **Just want a reliable one-shot? Run `sh saathratri-run-all-tests.sh`** (add `--regen` to regenerate
> first). It encodes every step and gotcha below — kills/frees ports, builds webapps, launches
> backends once, gates on real readiness, runs e2e judged by exit code, tears down, prints a
> per-app report, and exits non-zero on any failure. The manual steps below exist for when you
> need to run one phase in isolation or debug a failure. Flags: `--regen`, `--skip-backend`,
> `--skip-frontend`, `--no-e2e`. Logs land in `.test-logs/`.

---

## 0. The one discipline that prevents most pain

Every failure in this work came from **over-parallelizing tool calls** or **a backend port
collision**. Three rules:

1. **Run long/stateful commands one at a time.** Don't batch a regen + health-poll + e2e
   launch in one message — if one errors, the harness cancels the siblings and you lose track
   of what actually ran. Launch, wait for the notification, then proceed.
2. **Launch the 3 backends exactly once — AND free the ports first.** This is THE big one. The
   "cassandrastore e2e flake" that cost hours was never a Cypress timing issue: a second backend
   trio was launched while a prior one (or a straggler) still held 8080/8081/8082, so the new
   gateway died with `Port 8080 was already in use` → store's Cypress hit a dead `:8080`
   ("No request ever occurred" / "Cypress failed to verify your server"). **Before every launch:
   kill example java procs, kill whoever holds 8080/8081/8082, then BLOCK until those ports
   report zero listeners.** Killing-by-name alone is not enough — verify the ports actually
   cleared (a killed JVM can linger a few seconds). `saathratri-run-all-tests.sh` does this via
   `wait_ports_free`; do the same if running manually.
3. **Judge e2e by the Cypress exit code, not by grepping the log.** Cypress exits non-zero on
   any failed spec; log-scraping for "All specs passed" can match a stale line and report a
   false green. If you must summarize from the log, still gate pass/fail on `$?`.

## 1. Environment (Windows, behind TLS-intercepting proxy)

```bash
export NODE_OPTIONS=--use-system-ca                              # npm / ng / vitest / cypress / jhipster
export MAVEN_OPTS="-Djavax.net.ssl.trustStoreType=Windows-ROOT"  # maven (Windows trust store)
```

- **Docker must be running** — backend ITs use Cassandra 5 Testcontainers; e2e needs the
  docker-compose stack (Cassandra ×2, Postgres, JHipster Registry, Keycloak).
- Run backends/tests on **JDK 21** (the repo is built for 21).
- The bundled scripts (`saathratri-deploy.sh`, `saathratri-compile-dev.sh`) use `ttab` to open
  terminal tabs — **don't use them headlessly**; drive Maven/npm directly per the steps below.

## 2. Regenerate from the blueprint

The example is generated from `saathratri-apps-cassandra-mf.jdl` via the **npm-linked local**
`generator-jhipster-cassandra`. To pick up blueprint changes, link it first, then regen:

```bash
# once, so `jhipster --blueprints cassandra` resolves to the local working copy:
( cd ../generator-jhipster-cassandra && npm link )

# regen (from this repo root). NOTE: add --force — the bundled script omits it, which HANGS
# in a non-interactive shell on the "overwrite?" prompt:
cd <this repo>
jhipster --blueprints cassandra jdl saathratri-apps-cassandra-mf.jdl \
  --monorepository --workspaces --skip-jhipster-dependencies --force
```

Verify the link points at your working copy and is at the commit you expect:

```bash
# PowerShell:
(Get-Item (Join-Path (npm root -g) 'generator-jhipster-cassandra')).Target   # -> ...\workspace\generator-jhipster-cassandra
git -C ../generator-jhipster-cassandra rev-parse --short HEAD
```

A clean regen ends with `JHipster execution is complete`. The regen overwrites generated
files; a prior `git checkout -- . && git clean` of the example is optional but gives a clean diff.

## 3. Backend — unit + integration (all 3 apps)

Sequential, one app at a time (avoids Docker/RAM contention from parallel Cassandra containers):

```bash
for app in cassandrastore cassandrablog cassandragateway; do
  ( cd "$app" && ./mvnw -ntp -Dskip.npm clean verify > /tmp/be-$app.log 2>&1; echo "$app EXIT:$?" >> /tmp/be-$app.log )
done
```

Read results (first `Tests run` total per app = **surefire/unit**, second = **failsafe/IT**):

```bash
for app in cassandrastore cassandrablog cassandragateway; do
  echo "== $app =="; grep -E "^\[INFO\] Tests run: [0-9]+, Failures: [0-9]+, Errors: [0-9]+, Skipped: [0-9]+$" /tmp/be-$app.log
done
```

**Green baseline:** unit `store 33 / blog 58 / gateway 21`; IT `store 56 / blog 209 / gateway 40`;
all `Failures: 0, Errors: 0`. (Counts drift as entities change — what matters is 0 fail/err.)

The IT run includes the blueprint's vector work: `EmbeddingServiceTest`, the ANN AI-search IT,
the `CqlVector` round-trip IT, and `partialUpdate*ResourceIT` (guards the mapper null-ignore fix).

## 4. Frontend — vitest (all 3 apps)

`node_modules` must exist (run `npm install` per app once; slow, needs the proxy CA). Then:

```bash
for app in cassandragateway cassandrablog cassandrastore; do
  ( cd "$app" && npx ng test --watch=false > /tmp/fe-$app.log 2>&1; echo "$app EXIT:$?" >> /tmp/fe-$app.log )
done
# results:
for app in cassandragateway cassandrablog cassandrastore; do
  echo "== $app =="; sed -E 's/\x1b\[[0-9;]*m//g' /tmp/fe-$app.log | grep -E "Test Files|Tests +[0-9]+ (passed|failed)" | tail -2
done
```

**Green baseline:** `gateway 40 files/212`, `blog 117/625`, `store 47/217`; 0 failed. Includes
the composite-key search-form specs (blog) the blueprint adds.

## 5. E2E — Cypress (the orchestration-heavy part)

All three Cypress suites set `baseUrl: http://localhost:8080/` — they run against the **gateway
shell** with the remotes federated in. So the **whole stack must be up** first.

### 5a. Bring up the Docker stack (once)
The dev containers are usually already running. If not, per app:
`npm run docker:db:up` (Cassandra) in blog/store; gateway also needs Keycloak + Registry
(`docker compose -f src/main/docker/keycloak.yml up -d`, `... jhipster-registry.yml up -d`).
Confirm: `docker ps` shows `cassandrablog-cassandra`, `cassandrastore-cassandra`,
`cassandragateway-postgresql-1`, `...-keycloak-1`, `...-jhipster-registry-1` all healthy.

### 5b. Build each webapp into the backend (REQUIRED)
`spring-boot:run -Pdev` serves only the API — it does **not** compile the Angular client. Without
this, the post-login redirect to `/` returns **404 "No static resource index.html"** and every
spec dies in session setup. Build the client into `target/classes/static`:

```bash
for app in cassandragateway cassandrablog cassandrastore; do
  ( cd "$app" && npm run webapp:build )
done
# verify: each app's target/classes/static/index.html now exists
```

### 5c. Free the ports, THEN launch all 3 backends — ONE background job, never twice
First make absolutely sure nothing holds the server ports (see §0 rule 2 — this is the
store-"flake" root cause). Kill example java procs + whoever holds 8080/8081/8082, then block
until clear:
```bash
# kill by name AND by port, then wait until 0 listeners on 8080/8081/8082
powershell.exe -NoProfile -Command "
  Get-CimInstance Win32_Process -Filter \"Name='java.exe'\" |
    Where-Object { \$_.CommandLine -match 'jhipster-cassandra-example' -or \$_.CommandLine -match 'Cassandra(gateway|blog|store)App' } |
    ForEach-Object { try { Stop-Process -Id \$_.ProcessId -Force } catch {} }
  foreach (\$p in 8080,8081,8082) { Get-NetTCPConnection -State Listen -LocalPort \$p -EA SilentlyContinue | ForEach-Object { try { Stop-Process -Id \$_.OwningProcess -Force } catch {} } }"
until [ "$(powershell.exe -NoProfile -Command '(Get-NetTCPConnection -State Listen -LocalPort 8080,8081,8082 -EA SilentlyContinue | Measure-Object).Count' | tr -dc 0-9)" = 0 ]; do sleep 2; done
```
Then launch once:
```bash
export MAVEN_OPTS="-Djavax.net.ssl.trustStoreType=Windows-ROOT"
( ( cd cassandragateway && ./mvnw -ntp -Dskip.npm spring-boot:run -Dspring-boot.run.profiles=dev > /tmp/run-gateway.log 2>&1 ) &
  ( cd cassandrablog   && ./mvnw -ntp -Dskip.npm spring-boot:run -Dspring-boot.run.profiles=dev > /tmp/run-blog.log    2>&1 ) &
  ( cd cassandrastore  && ./mvnw -ntp -Dskip.npm spring-boot:run -Dspring-boot.run.profiles=dev > /tmp/run-store.log   2>&1 ) &
  wait )
```
If a backend log ends in `APPLICATION FAILED TO START` / `Port 80xx was already in use`, you
hit the collision — kill everything, free ports, relaunch. (Just use `saathratri-run-all-tests.sh`.)

### 5d. Wait until all 3 are healthy on the CORRECT ports, then let Eureka settle
```bash
until [ "$(curl -s -o /dev/null -w '%{http_code}' --max-time 4 http://localhost:8080/management/health)" = 200 ] \
   && [ "$(curl -s -o /dev/null -w '%{http_code}' --max-time 4 http://localhost:8081/management/health)" = 200 ] \
   && [ "$(curl -s -o /dev/null -w '%{http_code}' --max-time 4 http://localhost:8082/management/health)" = 200 ]; do sleep 10; done
curl -s -o /dev/null -w 'gateway root=%{http_code}\n' http://localhost:8080/    # expect 200 (index.html served)
sleep 30   # let the gateway register the remotes for module federation
```

### 5e. Run Cypress — use `e2e:cypress`, NOT `e2e:headless`
`npm run e2e` / `e2e:headless` chain a `ci:server:await` step that is **broken on Windows**
(`wait-on ... $npm_package_config_backend_port` — the var is never expanded → it waits forever
then fails). Run the cypress binary directly against the already-running stack:

```bash
for app in cassandragateway cassandrablog cassandrastore; do
  ( cd "$app" && npm run e2e:cypress > /tmp/e2e-$app.log 2>&1; echo "$app EXIT:$?" >> /tmp/e2e-$app.log )
done
for app in cassandragateway cassandrablog cassandrastore; do
  echo "== $app =="; grep -aE "All specs passed|of [0-9]+ failed" /tmp/e2e-$app.log | tr -d '\033' | sed 's/\x1b\[[0-9;]*m//g' | tail -1
done
```

**Green baseline:** gateway `11/11`, blog `150/150`, store `18/18` — "All specs passed!".

### 5f. Tear down the backends you started (leave the Docker stack)
```bash
# PowerShell:
Get-CimInstance Win32_Process -Filter "Name='java.exe'" |
  Where-Object { $_.CommandLine -match 'jhipster-cassandra-example' -or $_.CommandLine -match 'Cassandra(gateway|blog|store)App' } |
  ForEach-Object { Stop-Process -Id $_.ProcessId -Force }
```
Each app spawns **two** java procs (the Maven `spring-boot:run` launcher + the forked `…App`);
kill both. Confirm ports 8080/8081/8082 are free afterward.

## 6. Known gotchas (learned the hard way)

- **`e2e:headless` is broken on Windows** (unexpanded `$npm_package_config_backend_port`); use
  `npm run e2e:cypress` against a manually-started stack. See §5e.
- **Health-check 808x, not 934x/924x.** 9342/9242 are Cassandra CQL Docker ports. See §1.
- **Build the webapp before e2e** or you get the index.html 404. See §5b.
- **Port collision is the #1 e2e killer (the real "store flake").** A backend launched over a
  port still held by a prior/straggler JVM dies with `Port 80xx was already in use`; e2e then
  hits a dead `:8080` and reports `No request ever occurred` / `Cypress failed to verify your
  server`. It looks like a Cypress timing flake but it is NOT — it's a crashed backend. Always
  kill-by-name **and** kill-by-port, then **block until 8080/8081/8082 have zero listeners**
  before launching. See §0 rule 2 and §5c. Symptom check: `grep "APPLICATION FAILED TO START"
  /tmp/run-*.log`.
- **e2e pass/fail = Cypress exit code, never a log grep.** Grepping for "All specs passed" can
  match a stale line and falsely report green. See §0 rule 3 / §5e.
- **Single backend launch only.** Double-launch → `Port 8081 was already in use`. See §0/§5c.
- **`--force` on regen** or it hangs on the overwrite prompt. See §2.
- **Blueprint link staleness:** if a blueprint fix isn't showing up after regen, re-`npm link`
  and confirm the global symlink target + HEAD (§2). Generated specs are overwritten every
  regen — never hand-edit them as a "fix"; fix the blueprint template and regen.
- **Per-app asymmetry bites:** patches in the blueprint's cypress generator must not early-return
  on app shape. cassandrastore is **single-key-only** (no composite entities), which historically
  caused its specs to skip the cy.wait/intercept patches and time out. If only *some* apps' e2e
  fail, suspect an app-shape-conditional patch in `../generator-jhipster-cassandra/generators/cypress/generator.js`.

## 7. Quick reference (full run, in order)

**Preferred — the one-shot does all of the below, reliably:**
```bash
sh saathratri-run-all-tests.sh --regen      # regen + backend + frontend + e2e, ALL GREEN or non-zero exit
```

**Manual equivalent (use to run a single phase or debug):**
```bash
export NODE_OPTIONS=--use-system-ca MAVEN_OPTS="-Djavax.net.ssl.trustStoreType=Windows-ROOT"
( cd ../generator-jhipster-cassandra && npm link )                                   # link blueprint
jhipster --blueprints cassandra jdl saathratri-apps-cassandra-mf.jdl --monorepository --workspaces --skip-jhipster-dependencies --force
for a in cassandrastore cassandrablog cassandragateway; do (cd $a && ./mvnw -ntp -Dskip.npm clean verify); done   # backend+IT
for a in cassandragateway cassandrablog cassandrastore; do (cd $a && npx ng test --watch=false); done            # frontend
for a in cassandragateway cassandrablog cassandrastore; do (cd $a && npm run webapp:build); done                 # client into backend
# launch 3 backends (ONE job, §5c) → wait healthy on 8080/8081/8082 (§5d) → then:
for a in cassandragateway cassandrablog cassandrastore; do (cd $a && npm run e2e:cypress); done                  # e2e
# tear down backends (§5f); leave docker stack up
```
